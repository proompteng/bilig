@bilig/headless

Spreadsheet formulas for TypeScript services.

bilig is a small headless workbook runtime for code that needs formulas, edits, calculated values, and durable JSON state.

Use it from Node.js Build WorkPaper documents directly from TypeScript.
Check writes Edit an input, recalculate formulas, and read the result back.
Store the document Persist JSON in SQLite, Postgres, object storage, or your own database.
Install from npm Run the example Open starter issues Star on GitHub

Use a workbook when a table stops being enough.

A plain table can hold numbers. A WorkPaper can keep formulas, ranges, structural edits, and the serialized state that explains how a value was produced.

Formulas stay in the model

Create sheets, set formulas, edit inputs, and read calculated cells without opening a spreadsheet UI.

State can round-trip

Export a WorkPaper document, store it, restore it later, and keep the formulas and sheet structure intact.

Agents get readback

Tool-calling code can write a cell, recalculate, and return the value it actually produced.

Run a TypeScript smoke test.

This is enough to check formula calculation, writeback, serialization, and restore behavior from a clean Node project.

Create eval.ts.

The script throws if calculation or restore behavior changes. A good run prints verified: true.

import {
  WorkPaper,
  createWorkPaperFromDocument,
  exportWorkPaperDocument,
  parseWorkPaperDocument,
  serializeWorkPaperDocument,
} from "@bilig/headless";

type NumericCell = { value: number };

const workbook = WorkPaper.buildFromSheets({
  Revenue: [
    ["Region", "Customers", "ARPA", "Revenue"],
    ["West", 20, 1200, "=B2*C2"],
    ["East", 30, 250, "=B3*C3"],
    ["Central", 18, 300, "=B4*C4"],
  ],
  Summary: [
    ["Metric", "Value"],
    ["Total revenue", "=SUM(Revenue!D2:D4)"],
  ],
});

function numberValue(cell: unknown): number {
  if (typeof cell === "object" && cell !== null && typeof (cell as NumericCell).value === "number") {
    return (cell as NumericCell).value;
  }
  throw new Error(`Expected numeric cell value, got ${JSON.stringify(cell)}`);
}

function sheetId(name: string): number {
  const id = workbook.getSheetId(name);
  if (id === undefined) {
    throw new Error(`Missing sheet: ${name}`);
  }
  return id;
}

const before = numberValue(workbook.getCellValue({ sheet: sheetId("Summary"), row: 1, col: 1 }));
workbook.setCellContents({ sheet: sheetId("Revenue"), row: 1, col: 1 }, 32);

const saved = serializeWorkPaperDocument(exportWorkPaperDocument(workbook, { includeConfig: true }));
const restored = createWorkPaperFromDocument(parseWorkPaperDocument(saved));
const restoredSummary = restored.getSheetId("Summary");
if (restoredSummary === undefined) {
  throw new Error("Missing restored Summary sheet");
}

const after = numberValue(restored.getCellValue({ sheet: restoredSummary, row: 1, col: 1 }));
const verified = before === 36900 && after === 51300 && saved.length > 0;

if (!verified) {
  throw new Error(`Unexpected formula readback: ${JSON.stringify({ before, after, bytes: saved.length })}`);
}

console.log({ before, after, sheets: restored.getSheetNames(), bytes: saved.length, verified });

What the runtime is for.

bilig is useful when spreadsheet behavior belongs in code. It is not a finished Excel clone, and the comparison docs are clear about that boundary.

Capability Status
Programmatic workbook creation available
Formula evaluation and readback available
Structural row and column edits available
Snapshot, restore, and JSON persistence available

Good fit

  • Node services, serverless routes, and scripts that need spreadsheet formulas.
  • Agent tools that must verify a write before responding.
  • Formula-backed business logic that should be stored as a workbook document.
  • Internal tools where deterministic readback matters more than Excel UI parity.

Before you install

Can I use it without the browser app?

Yes. @bilig/headless is the package for Node.js scripts, services, and agent tools.

Is it a complete Excel clone?

No. It targets programmatic workbook workflows, and the compatibility docs call out current gaps.

How do I persist state?

Serialize the WorkPaper document and store it behind a small adapter. The docs show SQLite, Postgres, and object storage.

When is it a good fit?

When you need formulas, structural edits, deterministic readback, and JSON persistence from TypeScript.
